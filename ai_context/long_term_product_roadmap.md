# 糖錄錄長期產品路線草案

## 1. 長期願景

糖錄錄的長期願景可以是：

從 AI 語音血糖日記切入，逐步成為慢病管理平台加上個人健康資料平台，最後延伸成患者、家人、醫療院所與照護團隊共同使用的醫療資料協作平台。

長期目標是做得比智抗糖更全面。智抗糖有的能力，例如血糖、血壓、體重、飲食、運動、用藥、檢驗值、CGM、報告、家人共享、診所共享、AI 飲食、裝置串接，糖錄錄都應納入長期路線。

最大操作原則：

- 簡單操作。
- 能自動就自動。
- 能拍照就拍照。
- 能語音就語音。
- 手動輸入只作為兜底。
- 以長期 retention 與黏著度作為產品核心。

路線不應一開始就把所有功能做成完整成熟版本，而是先用非常清楚的小入口取得資料與信任，同時讓底層資料模型與報告架構能容納完整平台：

```text
血糖語音紀錄
-> 趨勢與報告
-> 醫師分享
-> 家人 / 照護者協作
-> 多慢病管理
-> 診所 / 醫院端平台
-> 醫療資料交換與照護網路
```

## 2. Phase 1：極簡血糖語音日記

目標：

- 驗證使用者是否願意用語音記錄血糖。
- 建立本地 AI 紀錄流程。
- 讓中老年人也能快速上手。
- 建立可擴充到慢病管理平台的資料骨架。

核心功能：

- 按住說話。
- 文字與手動輸入兜底。
- 本地 STT。
- 本地 LLM 結構化。
- 血糖、飲食、運動、用藥紀錄。
- 基本生活型態紀錄。
- 預留血壓、體重、檢驗值、CGM 的 generic health record。
- 使用者確認。
- SQLite encrypted local DB。
- 基本趨勢。
- 基本報告。

第一版可以簡化但要預留：

- 血壓、體重、檢驗值可以先用手動 / 語音基本紀錄，不一定先做裝置串接。
- CGM 可以先預留資料結構，不一定先串接。
- 醫院端可以先做分享報告，不一定先做完整後台。
- 拍照食物辨識可以先做資料結構與入口預留，完整 AI 辨識能力分階段上線。

## 3. Phase 2：趨勢、報告與回診準備

目標：

- 讓使用者看懂紀錄。
- 讓回診溝通變簡單。
- 把付費價值從「記錄」提升到「理解」。

核心功能：

- 7 / 14 / 30 天趨勢。
- 空腹 / 飯前 / 飯後 / 睡前分組。
- 飲食與運動事件標記。
- 回診摘要。
- PDF 報告。
- 一鍵分享。
- 單次授權連結 / QR code。
- 報告模板與資料來源抽象化。

差異化：

- 比智抗糖更簡單的醫師分享流程。
- 只讀、限時、可撤銷。
- 報告系統可擴充到血壓、體重、檢驗值、CGM 與醫院版摘要。

## 4. Phase 3：家人與照護者協作

目標：

- 讓慢病管理不只靠患者自己。
- 增加留存與日常使用價值。
- 支援家庭共享訂閱。

核心功能：

- 手機端快速切換 profile。
- 家人協助記錄。
- 照護提醒。
- 異常值通知。
- 每週摘要。
- 照護者留言。
- 授權管理。
- 家庭方案。
- 成員角色與權限。

注意：

- 家庭資料預設不共享。
- 家庭方案只共享訂閱權益。
- 預設不讓家人看到其他人的健康資料。
- 使用者可控制分享範圍。
- 所有存取寫 audit log。
- 訂閱共享與健康資料共享必須分開。
- 每筆紀錄都必須綁定 profile_id，避免照護者記錯對象。

## 5. Phase 4：多資料類型與裝置串接

目標：

- 從血糖擴到糖尿病相關健康資料。
- 降低手動輸入。
- 補齊智抗糖已有的主要健康資料範圍。

新增資料：

- 血壓
- 體重
- 體脂
- HbA1c
- 膽固醇
- 腎功能指標
- 步數
- 睡眠

裝置串接：

- Apple Health
- Health Connect
- 血糖機
- 血壓計
- 體重計
- CGM

策略：

- 第一版先預留資料模型。
- 第二階段先開手動 / 語音基本紀錄。
- 第三階段再依需求做裝置串接與自動同步。

## 6. Phase 5：診所 / 醫療人員入口

目標：

- 從個人 App 進入 B2B2C。
- 讓醫師、護理師、衛教師更容易看患者資料。

功能：

- 醫師檢視患者授權資料。
- 患者列表。
- 趨勢 dashboard。
- 回診前摘要。
- 異常值篩選。
- 訊息或提醒。
- 衛教內容派發。

權限：

- 只讀為預設。
- 診所內角色分權。
- 所有查看與匯出寫 audit log。

## 7. Phase 6：多慢病管理平台

目標：

- 從糖尿病擴到其他慢病。

可能擴展：

- 高血壓
- 高血脂
- 腎臟病
- 體重管理
- 心血管風險
- 代謝症候群

策略：

- 先從糖尿病自然相關資料擴張。
- 不要一次做所有病種。
- 每一種病種都要避免診斷與醫療處置建議。

## 8. Phase 7：醫療資料交換與平台化

目標：

- 成為患者授權的個人健康資料交換層。
- 支援美國與台灣市場的長期合規需求。

能力：

- FHIR-compatible data model。
- EHR / HIS 整合。
- 醫院 / 診所資料回寫或匯入。
- 患者授權中心。
- 大客戶 tenant 隔離。
- 合規 audit。
- 資料完整性驗證。

前提：

- HIPAA / PHI 控制成熟。
- 台灣個資與醫療資料合規審查完成。
- BAA 與第三方合規完成。
- 醫療合作模式明確。
- 法務與資安流程到位。

## 9. 長期護城河

糖錄錄可能的護城河：

- 極低摩擦的語音資料入口。
- 本地 AI 降低雲端成本。
- 長期個人健康資料。
- 漂亮且醫師可用的報告。
- 使用者授權資料共享網路。
- 醫療機構與照護團隊 workflow。
- 合規與資安能力。

## 10. 核心提醒

不要因為長期願景是醫療大平台，就在第一版把所有功能做完整成熟版本。

第一版真正要做到：

> 使用者講一句話，糖錄錄準確整理成血糖、飲食、運動、用藥與生活型態紀錄，並產生基本趨勢與報告；底層架構能一路擴到比智抗糖更全面的慢病管理與個人健康資料平台。

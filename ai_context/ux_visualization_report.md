# 糖錄錄 UX、趨勢圖與報告設計草案

## 1. 設計原則

糖錄錄第一版首頁要保持簡單，但不能讓產品看起來陽春。

核心原則：

- 首頁極簡，記錄流程最快。
- 趨勢頁美觀、清楚、容易看懂。
- 報告頁用純數據做出可信任感。
- 不用醫療化壓迫感，不用過多警告。
- 所有圖表都服務於理解，不做裝飾性複雜圖。

首頁負責輸入，趨勢頁負責理解，報告頁負責分享與回診。

## 2. 首頁

首頁元素：

- App 名稱：糖錄錄
- 今日摘要：今日已記錄幾筆
- 大型按住說話按鈕
- 文字輸入框
- 範例句
- 右上角功能選單

首頁不要放完整圖表，最多放一個小型摘要：

- 今日血糖最新值
- 最近 7 天平均
- 本週記錄天數

目的：

- 使用者一打開就知道要做什麼。
- 不讓中老年使用者被圖表、功能、文字淹沒。

## 3. 趨勢頁

趨勢頁要讓使用者快速看出：

- 最近血糖有沒有變高。
- 空腹和飯後是否不同。
- 哪些天記錄比較多。
- 飲食、運動、用藥和血糖是否有關聯。

第一版建議圖表：

### 7 / 14 / 30 天血糖趨勢

折線圖：

- X 軸：日期
- Y 軸：血糖值
- 點位顏色：空腹、飯前、飯後、睡前
- 可切換 7 天、14 天、30 天

避免：

- 一次顯示太多線。
- 用過多醫療警示色。
- 對使用者下診斷結論。

### 餐前餐後分布

簡單分組圖：

- 空腹平均
- 飯前平均
- 飯後平均
- 睡前平均

用途：

- 幫使用者理解不同測量情境。
- 不把不同類型血糖混成一個平均值誤導。

### 每日紀錄熱度

日曆 heatmap 或簡單日期格：

- 顏色深淺代表當日記錄數量。
- 不用讓使用者覺得沒記錄就是失敗。

### 飲食 / 運動事件標記

在血糖趨勢圖上用小圖示標記：

- 餐點
- 運動
- 用藥

點開後顯示當日紀錄摘要。

## 4. 報告功能

報告目標：

- 使用者自己看得懂。
- 回診時可以給醫師快速看。
- 純數據但排版美觀。
- 不做診斷，不給調藥建議。
- 從 MVP 開始就抽象化，未來擴充資料類型時不重寫核心。

報告範圍：

- 7 天
- 14 天
- 30 天
- 自訂日期區間

報告內容：

- 基本資訊：期間、總記錄數、記錄天數
- 血糖摘要：平均、最高、最低、空腹平均、飯後平均
- 趨勢圖：期間折線圖
- 分布圖：餐前 / 餐後 / 睡前
- 飲食摘要：常見食物文字列表
- 運動摘要：運動次數、總分鐘數
- 用藥紀錄：只列使用者記錄，不做建議
- 異常值列表：只列數值與時間，不做醫療判斷
- 原始紀錄附錄：可選

輸出格式：

- App 內報告頁
- PDF 匯出
- 分享連結，需授權與到期日

報告文案：

- 使用「紀錄摘要」、「趨勢」、「觀察」。
- 避免使用「診斷」、「治療建議」、「控制不良」等醫療判斷詞。

## 5. 報告引擎抽象化

報告功能是 MVP 必做，但第一版可以簡單。重點是底層架構要穩，未來新增血壓、體重、檢驗值、CGM、醫院版報告時不用重寫。

建議抽象：

### Data Provider

每一種資料類型提供標準介面：

- glucose provider
- meal provider
- exercise provider
- medication provider
- lifestyle provider
- blood pressure provider
- weight provider
- lab result provider
- CGM provider

第一版可以只實作前五個，其他先保留 interface。

### Metrics Calculator

負責把原始資料變成報告指標：

- 平均值
- 最高值
- 最低值
- 記錄天數
- 異常值列表
- 餐前餐後分組
- 趨勢變化
- 事件關聯

### Report Template

報告版型不直接綁死資料庫。

模板類型：

- user_summary
- doctor_visit
- caregiver_summary
- clinic_view
- premium_insight

### Renderer

同一份 report model 可輸出到不同呈現方式：

- App view
- PDF
- Web share page
- future clinic portal

### Export Policy

控制報告是否可下載、分享、過期、撤銷與 audit log。

欄位：

- export_id
- report_id
- owner_user_id
- viewer_scope
- expires_at
- revoked_at
- allow_download
- audit_required

### 設計結論

第一版報告可以只有血糖、飲食、運動、用藥、生活型態，但資料管線要從一開始支援更多 section。未來新增智抗糖既有功能或更進階醫療報告時，只新增 provider、calculator、template section，不重寫整套報告。

## 6. 醫院 / 醫師存取 UX

未來可支援使用者單次授權醫院或醫師存取資料。

授權方式：

- 使用者在 App 內產生一次性分享連結或 QR code。
- 使用者選擇資料範圍，例如最近 30 天。
- 使用者選擇有效時間，例如 24 小時或 7 天。
- 醫師開啟後只能看授權範圍內資料。
- 使用者可以隨時撤銷授權。

授權頁要清楚顯示：

- 誰可以看
- 可以看哪些資料
- 可以看到什麼時候
- 是否允許下載 PDF
- 是否允許匯出結構化資料

預設最小權限：

- 只讀
- 限定時間
- 限定資料範圍
- 不允許修改使用者資料

## 7. 圖表資料正確性

趨勢圖與報告不可直接讀未同步完成的衝突資料。

資料狀態應區分：

- local_only
- synced
- conflict
- deleted_pending_sync

報告預設只使用：

- 已確認資料
- 未被刪除資料
- 無衝突資料

若包含尚未同步資料，要清楚標示「此報告包含本機尚未同步紀錄」。
